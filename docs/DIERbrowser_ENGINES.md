# Почему не «копируем» Firefox/Chromium целиком в репозиторий

## Реальность open-source движков

| Исходники | Размер checkout | Типичная сборка | Кто так делает |
|-----------|-----------------|-----------------|----------------|
| **chromium/src** | десятки GB | часы, мощный CI | Google, Microsoft (Edge), Brave… |
| **firefox-main** (у вас в корне) | очень большой | `mach build`, специфичная среда | Mozilla |
| **CEF** (Chromium Embedded) | обёртка над Chromium | тяжёлая | Spotify, Discord (раньше), многие игры |

Сделать «полную копию браузера на C++» = по сути **стать дистрибутивом Chromium или Firefox**. Это отдельный продукт и команда на месяцы/годы, не один модуль в DierCHAT.

## Что мы используем вместо этого

**WebView2** (Windows) — официальный компонент Microsoft: внутри **тот же Chromium**, что и в Edge, обновляется вместе с **Edge WebView2 Runtime**.  
То есть вы **уже** используете наработки Chromium из открытого проекта (Blink/V8 и т.д.), просто не пересобираете их сами.

- Исходники **Edge** как готового продукта не «в открытом доступе», но **Chromium** открыт; WebView2 — поддерживаемый способ встроить этот класс движка в своё приложение.
- Папка **`firefox-main`** у вас остаётся **справочной** (спеки, идеи UX, будущая интеграция только если решите отдельный продукт на Gecko).

## Если когда-нибудь понадобится именно C++

Типичный путь не «форкнуть весь Firefox», а:

1. **CEF** — встраиваемый Chromium; оболочку (вкладки, UI) пишут на C++ или подключают к .NET/Java.
2. **Electron** — уже есть в DierCHAT; `<webview>` = Chromium.

Дублировать **и** полный UI на C++, **и** на C# для одного продукта обычно не делают: один клиент, один стек UI.

## Итог

**DIERbrowser** в репозитории — это **полнофункциональная оболочка браузера на C#** поверх **Chromium через WebView2**. Это соответствует ТЗ «как в полноценном браузере» по возможностям сайтов, без нереалистичной сборки всего дерева исходников у себя.

# Вход по email: «Ошибка запроса» / нет кода

## `503` на `POST /api/auth/verify` и текст «База данных недоступна»

**Да, причина именно в этом:** без рабочего **PostgreSQL** сервер не может завершить вход (создать/найти пользователя после кода). Код на почту мог уйти (SMTP + Redis), а **verify** вернёт **503**.

Что сделать: см. **`docs/LOCAL-SETUP.md`** — Docker `docker-compose.local.yml`, затем запуск Go с **`DIERCHAT_CONFIG=config.local.json`**, затем `npm run dev`.

Сообщение в консоли браузера про **CursorBrowser / Native dialog** к API и БД **не относится** — это настройка встроенного браузера Cursor.

---

## Что это значит

Клиент ходит на **`POST …/api/auth/send-code`**. Раньше при ответе не-JSON (или без поля `error`) показывалось общее **«Ошибка запроса»**. В новых сборках текст ошибки подробнее (сеть, HTTP-код, подсказка по API).

## Локальная разработка (браузер + Vite)

1. **Запустите DierCHAT-Server** (локально порт в **`config.local.json`**, сейчас **19080**; на проде — из `config.json`, часто **9000**).
2. Поднимите **PostgreSQL** и **Redis** так, как указано в `config.json` (хост/порт/пользователь). Без Redis сервер может стартовать, но хранение кода входа и часть функций будут работать некорректно.
3. В режиме **`npm run dev`** (порт **5173**) запросы к `/api` проксируются на **`http://127.0.0.1:19080`** (см. `vite.config.ts` и `config.local.json`).
4. Откройте клиент в **браузере** по адресу Vite (**5173**). Для прод-сборки или нестандартного API задайте **`VITE_API_BASE_URL`** (см. `src/lib/publicApiUrl.ts`).

Проверка вручную:

```bash
curl -s -X POST http://127.0.0.1:19080/api/auth/send-code -H "Content-Type: application/json" -d "{\"email\":\"test@example.com\"}"
```

Ожидается JSON вида `{"status":"code_sent"}`.

## Письмо с кодом не приходит

В логах сервера при старте предупреждение, если **SMTP** не настроен. Тогда код печатается **в консоль сервера**:

```text
[AUTH] Verification code for user@mail.com: 123456
```

Нужно заполнить **`smtp`** (или блок **`sms`** в старых конфигах — см. `DierCHAT-Server`) в `config.json`: host, login, password, from.

Ошибка отправки письма даёт от сервера сообщение вроде **«Ошибка отправки кода»** (HTTP 500) — смотрите лог сервера (`send-code: …`).

## «База данных недоступна» после ввода кода из письма

Код из email проверяется запросом **`POST /api/auth/verify`**. Для создания/поиска пользователя нужен **PostgreSQL**. Если при старте сервера в логе было *«PostgreSQL недоступен… Сервер запускается без базы данных»*, то **отправка кода** может работать (Redis + SMTP), а **вход по коду — нет**.

### Что сделать

1. Поднять **PostgreSQL** и **Redis** (проще всего через Docker из каталога сервера):

   Из **корня** репозитория (`DierCHAT`):

   ```bash
   docker compose -f docker-compose.local.yml up -d
   ```

   Либо из папки сервера:

   ```bash
   cd DierCHAT-Server
   docker compose -f docker-compose.local.yml up -d
   ```

2. Использовать конфиг, совместимый с этим compose: скопируйте **`DierCHAT-Server/config.local.json`** в рабочий `config.json` **или** поправьте в своём `config.json` блок `database`:
   - **порт `5435`** (на хосте; в контейнере по-прежнему 5432);
   - **пароль `dierchat_secure_pass`** (как в `docker-compose.local.yml`).

   Корневой `config.json` в репозитории часто задаёт порт **5432** и пароль **`dierchat`** — с контейнером из `docker-compose.local.yml` это **не совпадёт**, пока вы не создадите такого пользователя вручную или не смените настройки.

3. **Перезапустите DierCHAT-Server** с правильным путём к конфигу (`DIERCHAT_CONFIG` или `config.json` рядом с бинарником). В логе должно быть: **«PostgreSQL подключен»**.

4. При первом запуске сервер применяет миграции из `DierCHAT-Server/migrations/` (в Docker при старте контейера выполняется только `001_init.sql`; остальное — при старте Go-сервера, если БД доступна).

После этого снова запросите код и подтвердите.

## Продакшен (сайт)

Если открываете приложение с **вашего домена**, API берётся с **`window.location.origin`**. Убедитесь, что бэкенд развёрнут за тем же origin или задайте корректный **`VITE_API_BASE_URL`** при сборке фронта.

### `connectionfail` / «Страница не найдена» при открытии `http://IP:9000/`

Это **не ответ приложения**, а **нет TCP-соединения** с портом (сервис не слушает, фаервол, VDS выключен). С машины сервера проверьте: `curl -sS http://127.0.0.1:9000/api/health`. Снаружи часто нужно **`ufw allow 9000/tcp`** (и **80/tcp** для Nginx). Деплой: **`deploy-auto.bat`** и логи **`journalctl -u dierchat`**.

## Раздел 1: Авторизация и Управление аккаунтом
*(Исправлено согласно требованиям: "исправить способ входа", "сделать рассылку кода из моей собственной рабочей почты", "убрать вход по телефону")*

1. Полностью убрать вход по номеру телефона.
2. Реализовать вход только по email.
3. **Форма входа:**
    *   Поле для ввода email.
    *   Кнопка "Получить код".
    *   Валидация email (проверка формата).
4. **Бэкенд-логика отправки кода:**
    *   Генерация 6-значного числового кода.
    *   Сохранение кода в Redis (TTL - 5 минут, максимум 3 попытки ввода).
    *   Интеграция с SMTP-сервером вашей рабочей почты (указать хосты, порты, логин, пароль от почты).
    *   Формирование письма:
        *   Тема: `Код входа в Dier Chat`.
        *   Тело: "Ваш код: 123456. Никому не сообщайте его."
        *   Логотип Dier Chat (если есть).
5. **Форма ввода кода:**
    *   6 полей для цифр (автопереход к следующему полю).
    *   Таймер повторной отправки (30 секунд).
    *   Защита от брутфорса: после 5 неудачных попыток блокировка на 10 минут.
6. **Двухфакторная аутентификация (2FA):**
    *   Возможность установить дополнительный пароль в настройках.
    *   Запрос этого пароля после ввода кода (если включено).
    *   Сброс через email восстановления.
7. **Активные сессии:**
    *   Страница со списком устройств, где выполнен вход.
    *   Данные: браузер, ОС, IP-адрес, местоположение, время активности.
    *   Кнопка "Завершить все другие сессии".
8. **Автоматическое резервное копирование:** Настройки, темы, контакты синхронизируются с облаком.

**Как в Telegram:**
*   Используется MTProto для авторизации.
*   Код приходит в Telegram (или SMS). У нас — на email через SMTP.

## Раздел 2: Архитектура и Инфраструктура
*(Бэкенд-логика, обеспечивающая работу всего функционала)*

9. **Протоколы:**
    *   WebSocket (или аналог MTProto) для всего реалтайма.
    *   HTTPS (REST API) для тяжелых операций (загрузка файлов, история).
10. **База данных на клиенте (IndexedDB):**
    *   Все сообщения, контакты, чаты хранятся локально.
    *   При загрузке — читаем из кэша, потом догружаем новое.
11. **Diff-обновления:** Сервер присылает только изменения (не весь чат).
12. **Хранение на сервере:**
    *   **PostgreSQL:** Пользователи, чаты, сообщения, метаданные.
    *   **Redis:** Сессии, кэш, статусы онлайн.
    *   **Object Storage (S3/MinIO):** Фото, видео, файлы.
13. **End-to-End шифрование (секретные чаты):** Модуль с протоколом двойного ratchet (Signal Protocol).

**Как в Telegram:**
*   Используется MTProto v2 с мультиплексированием запросов.

## Раздел 3: Реалтайм синхронизация (WebSocket)
*(Исправление критического бага: "сообщения от собеседника появляются только после перезагрузки")*

14. **Установка соединения:**
    *   WebSocket соединяется сразу после авторизации.
    *   Адрес: `wss://api.dierchat.ru/ws?token=JWT_TOKEN`.
15. **Heartbeat (Пинг-Понг):**
    *   Каждые 30 сек клиент шлет `ping`, сервер отвечает `pong`.
    *   Нет ответа 10 сек — переподключение.
16. **Обработка событий:**
    *   Клиент слушает входящие события.
    *   Формат JSON: `{ "event": "new_message", "data": {...} }`.
17. **Список обязательных событий:**
    *   `new_message` — новое сообщение.
    *   `edit_message` — сообщение отредактировано.
    *   `delete_message` — сообщение удалено.
    *   `read_messages` — прочитано собеседником (галочки синие).
    *   `typing` — кто-то печатает.
    *   `online_status` — изменение статуса.
    *   `reaction_update` — новая реакция.
    *   `chat_action` — добавление/удаление из группы.
18. **Синхронизация после обрыва:**
    *   Запрос `sync_state` с `last_update_id`.
    *   Сервер возвращает пропущенные события.
19. **Индикатор соединения:**
    *   Иконка в интерфейсе: зеленая (ок), серая (нет сети), желтая (переподключение).
20. **Офлайн-очередь:**
    *   Если нет соединения, сообщения сохраняются локально.
    *   При появлении сети — автоматическая отправка.

**Как в Telegram (WebZ):**
*   Используется MTProto поверх WebSocket или Long Polling.
*   Сообщения доставляются за 0.2-0.5 сек.

# РАЗДЕЛ 23: LIQUID GLASS ИНТЕРФЕЙС (НОВЫЙ ДИЗАЙН TELEGRAM 2026)
*(Добавить согласно требованию: "Ликвид Гласс интерфейс распиши")*

## 23.1 Общее описание стиля Liquid Glass

**Liquid Glass** (Жидкое стекло) — это новая дизайн-концепция Telegram, внедренная в 2026 году . Основные характеристики:

143. **Эффект глубины и текучести:** Панели и элементы интерфейса выглядят как полупрозрачное "жидкое" стекло с динамическими бликами.
144. **Адаптивная прозрачность:** Элементы интерфейса меняют степень прозрачности в зависимости от контента под ними.
145. **Плавные анимации:** При скролле и переходах элементы "перетекают" друг в друга.
146. **Отсутствие жестких границ:** Вместо четких разделителей — мягкие тени и размытие (backdrop-filter).

## 23.2 Архитектурные требования для Liquid Glass

147. **Поддержка backdrop-filter:**
    *   Использовать CSS-свойства `backdrop-filter: blur()` и `backdrop-filter: brightness()` для создания эффекта стекла.
    *   Для браузеров без поддержки — fallback на полупрозрачные фоны (rgba).

148. **Динамическая адаптация:**
    *   При скролле списка чатов фон под панелями должен "переливаться" (параллакс-эффект).
    *   Плавающие панели (нижняя навигация, поле ввода) должны иметь усиленный эффект размытия.

149. **Настройки энергосбережения:**
    *   Добавить пункт "Анимация в чатах" → "Жидкое стекло" в настройках (раздел "Энергосбережение") .
    *   По умолчанию эффект может быть отключен для экономии батареи на мобильных устройствах.
    *   При включении — активировать все визуальные эффекты Liquid Glass.

## 23.3 Интерфейсные элементы в стиле Liquid Glass

150. **Нижняя навигационная панель (табы):**
    *   Плавающая панель (не прижата к краю экрана).
    *   Полупрозрачный фон с размытием (backdrop-filter: blur(20px)).
    *   Тонкая обводка (border) с rgba(255,255,255,0.2).
    *   Тень (box-shadow) с эффектом свечения .

151. **Поле ввода сообщения:**
    *   Не закреплено жестко к краям (отступы со всех сторон).
    *   Эффект "утопленного" стекла с внутренней тенью.
    *   Анимация расширения при фокусе (становится более "жидким") .

152. **Вкладки папок (Folders):**
    *   Скругленные плавающие кнопки над списком чатов.
    *   Активная вкладка — с эффектом подсветки (glow).

153. **Карточки чатов в списке:**
    *   Убрать разделительные линии между чатами (единое полотно) .
    *   При ховере/нажатии — легкое свечение (а не изменение фона).

154. **Модальные окна и попапы:**
    *   Глубокая прозрачность (фон сильно размыт).
    *   Плавное появление с масштабированием (scale + fade).

155. **Шапка профиля:**
    *   Компактные овальные кнопки ("Добавить", "Настройки", "QR-код") вместо прямоугольных.
    *   QR-код профиля в левом верхнем углу .

## 23.4 Анимации Liquid Glass

156. **При скролле:**
    *   Элементы слегка "деформируются" при прохождении через центр экрана.
    *   Скорость анимации привязана к скорости скролла (физика инерции).

157. **При открытии чата:**
    *   Анимация "перетекания" из списка в окно чата.
    *   Сообщения появляются с эффектом "волны" (последовательный fade-in).

158. **При отправке сообщения:**
    *   Облачко сообщения "вытекает" из поля ввода с эффектом натяжения.

159. **При переключении вкладок:**
    *   Контент сдвигается с параллакс-эффектом фона.

## 23.5 Цветовая схема Liquid Glass

160. **Светлая тема:**
    *   Основной фон: белый с легким градиентом.
    *   Стеклянные элементы: белый с прозрачностью 0.7-0.8, размытие 20px.
    *   Тени: холодные (синеватые), размытые.

161. **Темная тема:**
    *   Основной фон: глубокий черный (#0A0A0A).
    *   Стеклянные элементы: черный с прозрачностью 0.6, размытие 20px.
    *   Акценты: неоново-синие, фиолетовые блики.
    *   Обводки: rgba(255,255,255,0.1).

## 23.6 Техническая реализация

162. **CSS-код для стеклянной панели:**
```css
.glass-panel {
  background: rgba(255, 255, 255, 0.1);
  backdrop-filter: blur(20px) saturate(180%);
  -webkit-backdrop-filter: blur(20px) saturate(180%);
  border: 1px solid rgba(255, 255, 255, 0.2);
  border-radius: 24px;
  box-shadow: 0 8px 32px rgba(0, 0, 0, 0.1);
}
```

163. **Для темной темы:**
```css
.glass-panel-dark {
  background: rgba(10, 10, 10, 0.6);
  backdrop-filter: blur(20px);
  border: 1px solid rgba(255, 255, 255, 0.05);
  box-shadow: 0 8px 32px rgba(0, 0, 0, 0.4);
}
```

164. **JavaScript для определения поддержки:**
```javascript
const supportsBackdropFilter = 'backdropFilter' in document.documentElement.style || 
                               '-webkit-backdrop-filter' in document.documentElement.style;
if (!supportsBackdropFilter) {
  document.body.classList.add('no-glass');
}
```

165. **Анимация при скролле (Intersection Observer):**
    *   Использовать Intersection Observer для применения эффектов при появлении элементов.
    *   Коэффициент видимости влияет на прозрачность.

# РАЗДЕЛ 24: ДЕМОНСТРАЦИЯ ЭКРАНА С ВЫБОРОМ ЭКРАНА
*(Добавить согласно требованию: "демонстрацию экрана с выбором экрана распиши")*

## 24.1 Общее описание функции

166. **Назначение:** Возможность транслировать свой экран собеседнику во время видеозвонка или в отдельном режиме (как в Telegram).
167. **Поддержка множества источников:**
    *   Весь экран (монитор).
    *   Окно приложения.
    *   Вкладка браузера (для веб-версии).

## 24.2 API и технологии

168. **Основной API:** WebRTC Screen Capture API (getDisplayMedia) .
169. **Поддержка браузеров:**
    *   Chrome/Edge: полная поддержка.
    *   Firefox: полная поддержка.
    *   Safari: ограниченная поддержка (с 17 версии).
    *   Для неподдерживаемых браузеров — fallback-уведомление.

170. **Библиотеки:**
    *   Использовать adapter.js для кросс-браузерной совместимости .
    *   Для десктопных приложений (Electron) — использовать desktopCapturer.

## 24.3 Интерфейс выбора экрана

171. **Модальное окно выбора:**
    *   Запрос разрешения при первом использовании.
    *   Список доступных источников с превью (миниатюрами).
    *   Разделение на вкладки: "Экраны", "Окна", "Вкладки браузера".

172. **Дизайн окна выбора (Liquid Glass стиль):**
    *   Стеклянное модальное окно с размытием фона.
    *   Карточки источников с иконками и названиями.
    *   Превью в реальном времени (динамическое обновление).
    *   Кнопки: "Поделиться", "Отмена".

173. **Пример интерфейса (как в Telegram):**
    *   При нажатии на кнопку "Демонстрация экрана" в звонке.
    *   Появляется панель снизу (на мобилке) или модальное окно (на ПК).
    *   Пользователь видит список: "Экран 1 (1920x1080)", "Окно Chrome", "Эта вкладка".

## 24.4 Техническая реализация захвата

174. **Базовый код захвата:**
```javascript
async function startScreenCapture() {
  try {
    const stream = await navigator.mediaDevices.getDisplayMedia({
      video: {
        displaySurface: "monitor", // monitor, window, browser
      },
      audio: {
        suppressLocalAudioPlayback: false,
        systemAudio: "include" // включить системный звук
      },
      preferCurrentTab: false,
      selfBrowserSurface: "exclude", // исключить текущую вкладку
      surfaceSwitching: "include", // разрешить смену источника
      systemAudio: "include"
    });
    
    return stream;
  } catch (err) {
    console.error("Ошибка захвата экрана:", err);
    return null;
  }
}
```


175. **Опции захвата:**
    *   `displaySurface`: тип поверхности (monitor, window, browser).
    *   `surfaceSwitching`: разрешить переключение источника во время трансляции.
    *   `selfBrowserSurface`: показывать/скрывать текущую вкладку в списке.
    *   `systemAudio`: захватывать системный звук.

176. **Ограничения (constraints):**
```javascript
const constraints = {
  video: {
    width: { ideal: 1920 },
    height: { ideal: 1080 },
    frameRate: { ideal: 30 }
  }
};
```

## 24.5 Логика выбора нескольких мониторов

177. **Получение списка экранов:**
    *   На десктопе (Electron/C++): использовать системные API (DXGI на Windows, CGDisplay on macOS) .
    *   В браузере: getDisplayMedia сам предоставляет выбор пользователю.

178. **Для кастомного интерфейса выбора (как в Telegram Desktop):**
    *   На бэкенде использовать библиотеки для перечисления экранов (например, `screenshot` на Go) .
    *   В Qt/C++ использовать `QWebEngineDesktopMediaRequest::screensModel()` .

179. **Пример кастомного селектора (Electron):**
```javascript
const { desktopCapturer } = require('electron');

async function getScreenSources() {
  const sources = await desktopCapturer.getSources({
    types: ['screen', 'window'],
    thumbnailSize: { width: 320, height: 180 }
  });
  return sources;
}
```

## 24.6 Во время трансляции

180. **Индикатор трансляции:**
    *   Красная точка в статусе звонка.
    *   Плашка "Вы транслируете экран" с кнопкой "Остановить".

181. **Управление:**
    *   Кнопка "Пауза" (заморозить кадр).
    *   Кнопка "Переключить источник" (выбрать другое окно/экран).
    *   Кнопка "Остановить демонстрацию".

182. **Обработка остановки пользователем:**
    *   Системное уведомление, если пользователь вручную остановил через системные контролы.
    *   Автоматическое переключение на видео с камеры.

## 24.7 Производительность и качество

183. **Оптимизация потока:**
    *   Адаптивный битрейт в зависимости от пропускной способности сети.
    *   Максимальное разрешение: до 4K (при хорошем соединении).
    *   FPS: до 60 кадров/с.

184. **Сжатие:**
    *   Использовать кодек H264/VP9 для видео.
    *   Аппаратное ускорение кодирования (HW encode) где доступно.

185. **Для слабых сетей:**
    *   Понижение разрешения до 720p.
    *   Уменьшение FPS до 15.
    *   Приоритет звука над видео.

## 24.8 Мобильная версия (Android/iOS)

186. **На мобильных устройствах:**
    *   Захват всего экрана (через MediaProjection API на Android, ReplayKit на iOS).
    *   Предупреждение о безопасности (системное уведомление).
    *   Ограничение: нельзя захватить отдельное окно (только весь экран).

## 24.9 Безопасность и приватность

187. **Предупреждения:**
    *   При старте трансляции показывать предупреждение: "Вы будете транслировать экран. Личные данные могут быть видны собеседнику".
    *   В настройках приватности: "Кто может запрашивать демонстрацию экрана".

188. **Маскирование конфиденциальной информации:**
    *   Автоматически скрывать уведомления во время трансляции (опционально).
    *   Возможность включить "режим презентации" (скрыть личные чаты).

## 24.10 Интеграция со звонками (WebRTC)

189. **Добавление трека в существующий звонок:**
```javascript
// Получаем поток экрана
const screenStream = await startScreenCapture();

// Добавляем видео-трек в существующий peer connection
screenStream.getVideoTracks().forEach(track => {
  peerConnection.addTrack(track, screenStream);
});

// Удаляем трек камеры (опционально)
cameraStream.getVideoTracks().forEach(track => {
  track.stop();
});
```

190. **Переключение между камерой и экраном:**
    *   Кнопка "Переключить на камеру" во время трансляции.
    *   Замена видео-треков без переустановки соединения (replaceTrack).

23.1 Общее описание стиля Liquid Glass

Эффект глубины и текучести: Панели и элементы интерфейса выглядят как полупрозрачное "жидкое" стекло с динамическими бликами.

24.4 Техническая реализация захвата
Базовый код захвата:

javascript
async function startScreenCapture() {
  try {
    const stream = await navigator.mediaDevices.getDisplayMedia({
      video: {
        displaySurface: "monitor", // monitor, window, browser
      },
      audio: {
        suppressLocalAudioPlayback: false,
        systemAudio: "include" // включить системный звук
      },
      preferCurrentTab: false,
      selfBrowserSurface: "exclude", // исключить текущую вкладку
      surfaceSwitching: "include", // разрешить смену источника
      systemAudio: "include"
    });
    
    return stream;
  } catch (err) {
    console.error("Ошибка захвата экрана:", err);
    return null;
  }
}

Опции захвата:

displaySurface: тип поверхности (monitor, window, browser).

surfaceSwitching: разрешить переключение источника во время трансляции.

selfBrowserSurface: показывать/скрывать текущую вкладку в списке.

systemAudio: захватывать системный звук.

Фон становится серым/синим (в зависимости от темы).

Слева появляется кружок-чекбокс с галочкой.

Аватарка сообщения превращается в чекбокс.

Режим мультивыбора:

Верхняя панель меняется: отображается счетчик выбранных сообщений.

Кнопки действий: 🗑️ Удалить, 📤 Переслать, 📋 Копировать, ⭐ Сохранить в избранное.

Кнопка "Отмена" или "Закрыть" (крестик/стрелка назад).

Анимация:

Плавное появление чекбоксов.

Панель действий выезжает сверху (или снизу на мобилках).

25.4 Действия с выбранными сообщениями
Действие	Описание	Ограничения
Удалить	Удалить все выбранные сообщения (только свои или любые, если админ)	Подтверждение: "Удалить X сообщений?"
Переслать	Открыть список чатов → переслать все выбранные	Можно добавить комментарий к пересылке
Копировать	Скопировать текст всех сообщений в буфер обмена	Только текст (медиа не копируется)
Сохранить в избранное	Сохранить в чат "Избранное"	Каждое сообщение сохраняется отдельно
Закрепить	Закрепить выбранное сообщение (только одно)	При выборе >1 кнопка неактивна
Ответить	Ответить на выбранное сообщение (только одно)	При выборе >1 кнопка неактивна
25.5 Поведение при прокрутке
Сохранение выбора:

При скролле вверх/вниз выделенные сообщения остаются выделенными.

Подгрузка старых сообщений не сбрасывает выбор.

Выход из режима:

Нажать "Отмена" в верхней панели.

Нажать Escape (на ПК).

Выделить 0 сообщений (снять все).

25.6 Ограничения и особенности
Максимальное количество: Не более 100 сообщений за раз (для производительности).

Смешанные типы: Можно выделять текст, фото, видео, файлы, голосовые, стикеры.

Системные сообщения: Служебные сообщения (пользователь вошел, создал группу) нельзя выделить.

26.1 Аватарки — отображение в списке чатов и профилях
(Текущая проблема: аватарки видны только в профиле, но не в списке чатов и других местах)

Список чатов (левый сайдбар):

У каждого чата отображается круглая аватарка (40x40px).

Для личных чатов — аватарка собеседника.

Для групп — первая буква названия на цветном фоне, если нет общего фото.

Для каналов — аватарка канала.

Источник данных: Брать из БД/кэша, а не только из профиля текущего пользователя.

Окно чата (шапка):

Круглая аватарка (48x48px) рядом с именем собеседника/группы.

При клике — открывается профиль с большой аватаркой.

Список участников группы:

У каждого участника — мини-аватарка (32x32px).

Рядом с именем.

Уведомления (push):

В системном уведомлении отображается аватарка отправителя (если поддерживается платформой).

Реакции под сообщениями:

При нажатии на реакцию — показывать мини-аватарки пользователей, которые её поставили (всплывающее окно).

Кэширование аватарок:

Хранить в IndexedDB.

Обновлять при изменении аватара пользователем (через WebSocket событие avatar_update).

26.2 Истории — видимость для других пользователей
(Текущая проблема: истории видит только автор, другие пользователи не видят)

Публикация истории:

При создании истории отправлять её на сервер, а не хранить локально.

Сервер сохраняет историю в БД с привязкой к пользователю.

TTL (время жизни) — 24 часа, затем автоматическое удаление.

Отображение у других пользователей:

При загрузке списка чатов делать запрос на сервер: GET /stories?user_id=... для всех контактов.

Если у контакта есть активная история (менее 24 часов), вокруг аватарки отображается цветное кольцо (синее/градиентное).

При клике на кольцо — открывается просмотр истории.

Кольца аватарок в списке чатов:

Порядок: сначала те, у кого есть непросмотренная история (сверху).

Анимация пульсации кольца для новых историй.

Просмотр истории:

Открывается на весь экран.

Свайп влево/вправо — переключение между историями разных пользователей.

Свайп вниз — закрыть.

Тап по левой стороне — предыдущий кадр, по правой — следующий.

Статус просмотра:

Сервер фиксирует, кто просмотрел историю.

Автор видит список просмотревших (глазок + счетчик).

26.3 Каналы — комментарии
(Текущая проблема: в канале нет комментариев)

Архитектура комментариев в каналах:

Каждый пост в канале может иметь привязанную группу для комментариев.

Создание отдельной группы-комментатора автоматически при создании канала (опционально).

Интерфейс:

Под каждым постом в канале кнопка "💬 Комментарии (12)".

При нажатии открывается окно чата с группой-комментатором.

Комментарии отображаются как обычные сообщения в группе.

Для существующих каналов:

Возможность привязать существующую группу к каналу.

Возможность создать новую группу для комментариев.

Права:

Админ канала может управлять комментариями (отключить, удалить).

Владелец группы-комментатора может модеррировать.

26.4 Видео и Видеосообщения (кружки) — разделение функционала
(Текущая проблема: видео сворачивается в кружок, нужно разделить)

26.4.1 Обычное видео (не кружок)
Отправка видео:

Пользователь выбирает видео из галереи или загружает файл.

Видео отображается в сообщении как прямоугольный плеер с соотношением сторон оригинала.

Не обрезается, не сворачивается в круг.

Плеер для видео:

Встраивается в сообщение (ширина 100%, высота авто).

Элементы управления: пауза/воспроизведение, ползунок времени, громкость, полноэкранный режим.

Кнопка скачивания (опционально).

Поведение при клике:

При клике на видео открывается полноэкранный режим.

Свайп/стрелки — переключение между медиа в чате.

26.4.2 Видеосообщения (кружки)
Отправка видеосообщения:

Кнопка отдельная (круглая иконка рядом с полем ввода).

Запись с камеры (MediaRecorder) длительностью до 60 секунд.

Формат — квадратное или круглое видео.

Визуальное оформление:

Круглая маска (border-radius: 50%).

Размер 120x120px (как в Telegram).

При наведении/клике — увеличение до 180x180px.

Специальный плеер для кружков (как в Telegram):

Обводка (прогресс-бар по кругу): Вокруг кружка отображается кольцевой прогресс воспроизведения.

Круговая перемотка: При клике и движении по кругу — перемотка назад/вперед (вращение пальцем по кругу).

Пауза: Одиночный клик по кружку — пауза/воспроизведение.

Скорость воспроизведения: Долгий тап — выбор скорости (x1, x1.5, x2).

Техническая реализация круговой перемотки:

javascript
// Пример логики круговой перемотки
const handleCircularSeek = (event, videoElement, radius, centerX, centerY) => {
  const angle = Math.atan2(event.clientY - centerY, event.clientX - centerX);
  const progress = (angle + Math.PI) / (2 * Math.PI); // от 0 до 1
  videoElement.currentTime = progress * videoElement.duration;
  updateCircleProgress(progress);
};
26.5 Звонки — двустороннее соединение
(Текущая проблема: при звонке окно появляется у звонящего, но не у получателя)

Архитектура WebRTC звонков:

Сигналинг через WebSocket (обмен SDP и ICE-кандидатами).

При входящем звонке сервер отправляет событие incoming_call целевому пользователю.

Обработка входящего звонка (на стороне получателя):

javascript
socket.on('incoming_call', (data) => {
  // Показать модальное окно с аватаркой, именем, кнопками "Принять"/"Отклонить"
  showCallModal({
    caller: data.from_user,
    callId: data.call_id,
    onAccept: () => acceptCall(data.call_id),
    onReject: () => rejectCall(data.call_id)
  });
  
  // Воспроизвести звук входящего звонка (ringtone)
  playRingtone();
});
Условия получения звонка:

Получатель должен быть онлайн (WebSocket соединение активно).

Если получатель офлайн — звонок не проходит, отображать "Пользователь не в сети".

Интерфейс звонка:

У звонящего: "Вызов...", аватарка собеседника, кнопка "Отмена".

У получателя: модальное окно с аватаркой, именем, кнопками "Принять"/"Отклонить".

ICE-серверы (STUN/TURN):

Настроить публичные STUN-серверы (Google: stun:stun.l.google.com:19302).

Для пробоя NAT — TURN-сервер (собственный или платный).

26.6 Стикеры — серверное хранение
(Текущая проблема: стикеры хранятся локально, не синхронизируются между устройствами)

Архитектура:

Стикеры хранятся на сервере (Object Storage S3/MinIO).

Каждый стикер имеет ID, набор, автора, эмодзи-теги.

При отправке стикера передается только ID, клиент подгружает URL.

Загрузка пользовательских стикеров:

Интерфейс загрузки PNG/JPG/MP4.

Обрезка фона (опционально через API remove.bg).

Загруженный стикер отправляется на сервер, сохраняется в S3, привязывается к пользователю.

Синхронизация между устройствами:

При входе с нового устройства — список стикеров подгружается с сервера.

Добавление/удаление стикера синхронизируется через WebSocket.

Магазин стикеров:

Список публичных наборов.

Кнопка "Установить" — добавляет набор в библиотеку пользователя.

26.7 Музыка и Голосовые сообщения — разделение
(Требование: музыка и голосовые — это разное, сделать отдельный бар для управления треком)

26.7.1 Голосовые сообщения (не меняется)
Голосовые сообщения:

Отображаются в сообщении как компактный блок с визуализацией (волны).

Управление: пауза/воспроизведение, скорость (x1, x1.5, x2).

Не имеют отдельного плеера в шапке.

26.7.2 Музыка (аудиофайлы, треки)
Отправка аудио:

Файлы формата MP3, M4A, AAC, FLAC.

Отображаются в сообщении с обложкой (если есть в метаданных) или стандартной иконкой.

Глобальный музыкальный плеер (бар управления):

При воспроизведении любого аудио (кроме голосовых) появляется плавающий бар внизу экрана (как в Telegram).

Содержимое бара:

Обложка трека (или иконка).

Название трека (имя файла).

Исполнитель (если есть ID3-теги).

Кнопки: пауза/воспроизведение, следующий трек (если есть очередь), закрыть.

Ползунок прогресса.

Бар остается при переключении между чатами.

Очередь воспроизведения:

При добавлении нескольких аудио — они становятся в очередь.

Кнопка "Следующий" переключает на следующий трек.

Логика разделения:

javascript
// Определение типа аудио
if (message.media_type === 'voice') {
  renderVoiceMessage(); // компактный плеер в сообщении
} else if (message.media_type === 'audio') {
  renderAudioMessage(); // сообщение с кнопкой "Добавить в плеер"
  addToGlobalPlayer(message); // добавляем в глобальный бар
}
Поведение:

Голосовые НЕ попадают в глобальный плеер.

Музыка (аудиофайлы) попадают в глобальный плеер.

При клике на аудио в сообщении — оно начинает играть в глобальном плеере.

РАЗДЕЛ 27: ИСПРАВЛЕНИЕ ПРОФИЛЯ ПОЛЬЗОВАТЕЛЯ — АДАПТИВНАЯ ВЕРСТКА

27.1 Описание проблемы
Текущее состояние:

На десктопной версии профиль отображается корректно (центрирован, ширина нормальная).

На мобильных устройствах (iPhone/Android) страница профиля не адаптируется под экран:

Контент прижат к левому краю.

Ширина контейнера меньше ширины экрана.

Аватарка и текст не центрированы.

Элементы управления (кнопки) съезжают.

Требуемое состояние:

Страница профиля должна корректно отображаться на любых устройствах:

Десктоп (ширина > 1024px) — центрированная карточка профиля.

Планшет (768px - 1024px) — адаптивная ширина.

Мобильные (< 768px) — полноэкранный режим с отступами.

# РАЗДЕЛ 28: ВСТРОЕННЫЙ БРАУЗЕР (IN-APP BROWSER)
*(Полнофункциональный браузер внутри мессенджера для открытия ссылок без выхода из приложения)*

---

## 28.1 Общее описание

255. **Назначение:** При клике на любую ссылку в сообщении открывается встроенный браузер внутри Dier Chat, а не внешний браузер системы.

256. **Где используется:** Telegram, WhatsApp, Viber, Slack.

257. **Преимущества:**
    - Пользователь не покидает приложение.
    - Сохраняется контекст чата (можно быстро вернуться).
    - Безопасность (контроль над открываемыми сайтами).
    - Возможность авторизации через мессенджер (Telegram Login).

---

## 28.2 Запуск встроенного браузера

258. **Способы открытия:**
    - Клик по любой ссылке в сообщении (http, https).
    - Клик по превью ссылки (link preview).
    - Клик по кнопке "Открыть в браузере" в контекстном меню ссылки.
    - Из меню чата (опционально).

259. **Анимация открытия:**
    - Плавное выезжание снизу (на мобилках) или масштабирование (на десктопе).
    - Затемнение фона чата.
    - Кнопка закрытия (крестик) в правом верхнем углу.

---

## 28.3 Интерфейс встроенного браузера

260. **Верхняя панель (навигация):**
    - Кнопка "Назад" (←) — возврат на предыдущую страницу.
    - Кнопка "Вперед" (→) — переход вперед по истории.
    - Кнопка "Обновить" (⟳) — перезагрузка страницы.
    - Кнопка "Поделиться" (↗) — поделиться ссылкой в чате.
    - Кнопка "Закрыть" (✕) — закрыть браузер, вернуться в чат.

261. **Адресная строка (URL bar):**
    - Отображение текущего URL.
    - Возможность редактирования и перехода по новому URL.
    - Индикатор безопасности (замок для HTTPS, предупреждение для HTTP).
    - Прогресс загрузки страницы (полоска под адресной строкой).

262. **Нижняя панель (опционально, мобилки):**
    - Кнопки: Назад, Вперед, Обновить, Поделиться, Закрыть (для удобства одной рукой).

263. **Контекстное меню (долгий тап на странице):**
    - Копировать ссылку.
    - Открыть в системном браузере.
    - Поделиться.
    - Найти на странице.

---

## 28.4 Функциональные возможности

### 28.4.1 Навигация

264. **История посещений:**
    - Хранится в рамках текущей сессии браузера.
    - Кнопки "Назад"/"Вперед" работают корректно.
    - При закрытии браузера история очищается.

265. **Загрузка страниц:**
    - Отображение индикатора загрузки (спиннер).
    - Таймаут загрузки (30 секунд) с предложением перезагрузить или открыть в системном браузере.

266. **Обработка ошибок:**
    - Страница "Не удалось загрузить сайт" с кнопками:
        - "Повторить"
        - "Открыть в браузере"
        - "Закрыть"

### 28.4.2 Безопасность

267. **HTTPS/SSL:**
    - Визуальное отличие (зеленый замок для HTTPS, серый для HTTP).
    - Предупреждение при переходе на HTTP с HTTPS.

268. **Блокировка опасных сайтов:**
    - Интеграция с сервисом проверки ссылок (Google Safe Browsing API).
    - При попытке открыть фишинговый/вредоносный сайт — красное предупреждение.

269. **Защита от слежения:**
    - Отключение сторонних cookies (опционально в настройках).
    - Блокировка трекеров (как в Telegram).

### 28.4.3 Мультимедиа

270. **Воспроизведение видео:**
    - HTML5 video поддерживается нативно.
    - Полноэкранный режим с поворотом экрана (на мобилках).

271. **Аудио:**
    - Воспроизведение в фоне (если браузер закрыт, аудио останавливается).

272. **Скачивание файлов:**
    - При клике на ссылку на файл — предложение:
        - "Скачать"
        - "Сохранить в Избранное"
        - "Открыть во внешнем браузере"

### 28.4.4 Интеграция с мессенджером

273. **Telegram/Dier Chat Login:**
    - Поддержка виджетов авторизации через мессенджер.
    - При нажатии на кнопку "Login with Telegram" — автоматическая авторизация через текущий аккаунт.

274. **Открытие ссылок из браузера в чат:**
    - Кнопка "Поделиться" → открывает окно выбора чата → вставляет ссылку с превью.

275. **Deep links:**
    - Поддержка ссылок вида `https://t.me/username` или `dierchat://chat?id=123`.
    - При открытии таких ссылок — автоматический переход в соответствующий чат (без подтверждения).

## 28.5 Техническая реализация

276. **WebView компонент (React/Vue):**
```jsx
// InAppBrowser.jsx
import { useState, useRef } from 'react';

export function InAppBrowser({ url, onClose }) {
  const [currentUrl, setCurrentUrl] = useState(url);
  const [canGoBack, setCanGoBack] = useState(false);
  const [canGoForward, setCanGoForward] = useState(false);
  const [isLoading, setIsLoading] = useState(true);
  const webviewRef = useRef(null);

  const goBack = () => {
    if (webviewRef.current && canGoBack) {
      webviewRef.current.goBack();
    }
  };

  const goForward = () => {
    if (webviewRef.current && canGoForward) {
      webviewRef.current.goForward();
    }
  };

  const reload = () => {
    if (webviewRef.current) {
      webviewRef.current.reload();
    }
  };

  const shareLink = () => {
    // Открыть окно выбора чата для отправки ссылки
    openShareDialog(currentUrl);
  };

  return (
    <div className="inapp-browser">
      <div className="browser-header">
        <button onClick={goBack} disabled={!canGoBack}>←</button>
        <button onClick={goForward} disabled={!canGoForward}>→</button>
        <button onClick={reload}>⟳</button>
        <div className="url-bar">
          <span className="security-icon">
            {currentUrl.startsWith('https') ? '🔒' : '⚠️'}
          </span>
          <input
            type="text"
            value={currentUrl}
            onChange={(e) => setCurrentUrl(e.target.value)}
            onKeyPress={(e) => e.key === 'Enter' && navigateTo(currentUrl)}
          />
        </div>
        <button onClick={shareLink}>↗</button>
        <button onClick={onClose}>✕</button>
      </div>
      
      {isLoading && <div className="loading-spinner">Загрузка...</div>}
      
      <iframe
        ref={webviewRef}
        src={url}
        className="browser-content"
        onLoadStart={() => setIsLoading(true)}
        onLoad={() => setIsLoading(false)}
        sandbox="allow-same-origin allow-scripts allow-popups allow-forms allow-modals"
      />
    </div>
  );
}
```

290. **Индикация:**
    - В чате, при клике на ссылку, кнопка "Открыть в браузере" подсвечивается.

    - Поворот экрана (мобилки).

# РАЗДЕЛ 29: УПРАВЛЕНИЕ ИСТОРИЯМИ (STORIES MANAGEMENT)
*(Полный функционал управления историями: создание, редактирование, удаление, настройки приватности, как в Telegram)*

---

## 29.1 Общее описание

292. **Назначение:** Возможность создавать, редактировать, управлять и удалять истории (Stories) — временные публикации, которые исчезают через 24 часа.

293. **Где используется:** Telegram, Instagram, WhatsApp, Snapchat.

294. **Основные возможности:**
    - Публикация фото/видео на 24 часа.
    - Редактирование текста, рисование, добавление стикеров.
    - Управление приватностью (кто видит историю).
    - Просмотр статистики (кто посмотрел).
    - Удаление истории в любой момент.

---

## 29.2 Создание истории

### 29.2.1 Способы создания

295. **Кнопка добавления:**
    - В списке чатов — кнопка "+" над аватаркой пользователя.
    - В профиле — кнопка "Добавить историю".
    - Долгий тап на аватарке в шапке чата.

296. **Выбор источника:**
    - Камера (фото/видео) — запись прямо в приложении.
    - Галерея (выбор существующих фото/видео).
    - Файловая система (для ПК).

### 29.2.2 Редактор историй (Story Editor)

297. **Интерфейс редактора (как в Telegram):**
    - Предпросмотр истории на весь экран.
    - Инструменты внизу экрана.

298. **Инструменты редактирования:**

| Инструмент | Описание | Как в Telegram |
|------------|----------|----------------|
| **Текст** | Добавление текста поверх медиа | Шрифты: обычный, моноширинный, серифный; выравнивание; цвет |
| **Рисование** | Кисть/маркер для рисования | Толщина кисти, палитра цветов, ластик |
| **Стикеры** | Добавление стикеров из набора | Перетаскивание, масштабирование, поворот |
| **Эмодзи** | Добавление эмодзи поверх | Поиск эмодзи, последние использованные |
| **Обрезка** | Обрезка видео/фото | Кадрирование, соотношение сторон |
| **Звук** | Добавление музыки к видео | Выбор трека, обрезка, громкость |

### 29.2.3 Работа с текстом

299. **Функции текста:**
    - Разные шрифты (3-4 варианта).
    - Размер текста (регулируется пинчем).
    - Цвет текста (палитра).
    - Фон текста (полупрозрачный, круглый, прямоугольный).
    - Выравнивание (по левому/правому краю, по центру).

300. **Пример интерфейса текста:**
```javascript
// Компонент добавления текста
const addTextToStory = (text, style) => {
  const textOverlay = {
    text: text,
    font: style.font, // 'default', 'serif', 'mono'
    size: style.size,
    color: style.color,
    background: style.background, // 'none', 'rounded', 'rect'
    position: { x: 50, y: 50 },
    rotation: 0,
    scale: 1
  };
  storyEditor.addOverlay(textOverlay);
};
```

### 29.2.4 Работа со стикерами

301. **Добавление стикеров:**
    - Доступны все наборы стикеров пользователя.
    - Поиск стикеров по эмодзи.
    - Поддержка анимированных стикеров (MP4/WebM).

302. **Управление стикером:**
    - Перетаскивание (drag & drop) в любую точку экрана.
    - Пинч для масштабирования.
    - Двойной тап для удаления.
    - Поворот двумя пальцами.

### 29.2.5 Добавление музыки (для видео)

303. **Выбор трека:**
    - Из библиотеки Telegram (популярные треки).
    - Из медиатеки пользователя.
    - Поиск по названию/исполнителю.

304. **Настройки звука:**
    - Обрезка трека (выбор фрагмента).
    - Громкость музыки.
    - Оригинальный звук видео (можно выключить).

---

## 29.3 Публикация истории

305. **Кнопка "Опубликовать" (галочка внизу).**

306. **Настройки перед публикацией:**
    - **Приватность:** Кто может видеть историю.
    - **Сохранение в галерею:** Сохранить оригинал на устройство.
    - **Добавить в профиль:** (всегда да).

307. **Процесс публикации:**
    - Медиа загружается на сервер.
    - Прогресс загрузки (индикатор).
    - После загрузки — история появляется у подписчиков.
    - Уведомление "История опубликована".

---

## 29.4 Приватность историй

308. **Уровни приватности:**
    - **Все** — видят все пользователи (по умолчанию).
    - **Контакты** — только пользователи в контактах.
    - **Близкие друзья** — выбранный список.
    - **Никто (скрыть)** — никто не видит (фактически отложенная публикация).

309. **Настройка "Близких друзей":**
    - Специальный список контактов в настройках.
    - Можно добавлять/удалять пользователей.
    - Синхронизируется между устройствами.

310. **Исключения:**
    - **Скрыть от пользователей:** конкретные контакты, которые не увидят историю.
    - **Показать только для:** конкретные контакты, которые увидят историю.

311. **Сохранение настроек:**
    - Последняя использованная настройка приватности запоминается для следующей истории.

---

## 29.5 Просмотр и управление опубликованными историями

### 29.5.1 Отображение в списке чатов

312. **Кольцо вокруг аватарки:**
    - Синее/градиентное кольцо — есть непросмотренная история.
    - Серое кольцо — история просмотрена.
    - Анимация пульсации — новые истории.

313. **Порядок отображения:**
    - Сверху — пользователи с непросмотренными историями.
    - Далее — остальные с историями.
    - Свой профиль — всегда в начале.

### 29.5.2 Просмотр своей истории

314. **Клик на свою аватарку:**
    - Открывается просмотр своей истории.
    - Кнопка "Добавить историю" (если нет активной).
    - Если есть активная история — просмотр.

315. **Интерфейс просмотра (своя история):**
    - Прогресс-бар вверху.
    - Счетчик просмотров (глазок + число).
    - Список просмотревших (при клике на глазок).
    - Кнопка "..." (меню управления).

### 29.5.3 Просмотр чужих историй

316. **Интерфейс:**
    - Круговая анимация прогресса вокруг аватарки при просмотре.
    - Свайп влево/вправо — переключение между историями разных пользователей.
    - Свайп вниз — закрыть.
    - Тап по левой стороне — предыдущий кадр (если несколько в истории).
    - Тап по правой стороне — следующий кадр.

317. **Реакции на истории:**
    - Быстрые реакции (👍❤️😂😮😢🔥👎) — внизу экрана.
    - Отправка реакции автору (уведомление).
    - Реакции отображаются в просмотревших.

318. **Ответ на историю:**
    - Кнопка "Ответить" (💬) — открывает чат с автором с превью истории.
    - Поле ввода с цитатой истории.

---

## 29.6 Управление историей (меню)

319. **Меню управления (три точки):**
    - Открывается при просмотре своей истории.
    - Или при долгом нажатии на свою аватарку в списке.

320. **Пункты меню:**

| Пункт | Описание |
|-------|----------|
| **Удалить** | Удалить историю (немедленно, без подтверждения для всех) |
| **Редактировать** | Открыть редактор для внесения изменений |
| **Изменить приватность** | Изменить настройки видимости (кто видит) |
| **Сохранить в галерею** | Сохранить оригинал на устройство |
| **Статистика** | Просмотреть список просмотревших |
| **Закрепить в профиле** | Закрепить историю в профиле (не исчезает через 24 часа) |

### 29.6.1 Редактирование истории

321. **Возможность редактирования после публикации:**
    - Изменить текст/рисунок/стикеры.
    - Изменить приватность.
    - Добавить/удалить медиа (если несколько в истории).

322. **Обновление истории:**
    - При редактировании — версия обновляется для всех.
    - Просмотревшие не сбрасываются.
    - Уведомление "История обновлена" (для автора).

### 29.6.2 Удаление истории

323. **Способы удаления:**
    - Из меню истории (три точки → Удалить).
    - Свайп вверх в режиме просмотра (как в Telegram).
    - Автоматическое удаление через 24 часа.

324. **Подтверждение удаления:**
    - Диалог: "Удалить историю? Это действие нельзя отменить."
    - Кнопки: "Удалить", "Отмена".

325. **Удаление нескольких историй:**
    - Режим мультивыбора в архиве историй.
    - Выбрать несколько → кнопка "Удалить".

---

## 29.7 Статистика просмотров

326. **Счетчик просмотров:**
    - Отображается под историей при просмотре.
    - Формат: "👁️ 123 просмотра".

327. **Список просмотревших (при клике на счетчик):**
    - Аватарка, имя, username.
    - Время просмотра (только для автора).
    - Реакции (если поставили).

328. **Формат списка:**
```javascript
const viewersList = [
  { id: 1, name: "Анна", username: "anna", avatar: "...", viewedAt: "10:30", reaction: "❤️" },
  { id: 2, name: "Иван", username: "ivan", avatar: "...", viewedAt: "10:32", reaction: null },
  // ...
];
```

329. **Сортировка:**
    - Сначала — друзья (контакты).
    - Затем — остальные.
    - По времени просмотра (новые сверху).

---

## 29.8 Архив историй

330. **Доступ к архиву:**
    - В профиле пользователя — раздел "Архив историй".
    - Иконка календаря/часов.

331. **Отображение архива:**
    - Сетка из миниатюр всех опубликованных историй (за все время).
    - Сортировка по дате (новые сверху).
    - Статистика просмотров под каждой миниатюрой.

332. **Действия в архиве:**
    - Просмотр истории (как в момент публикации).
    - Восстановление (если нужно показать снова).
    - Удаление навсегда.
    - Скачивание.

## 29.9 Техническая реализация

333. **Структура данных (сервер):**
```javascript
// Модель Story
{
  id: "story_123456",
  userId: "user_789",
  mediaType: "photo" | "video",
  mediaUrl: "https://cdn.dierchat.ru/stories/...",
  thumbnailUrl: "https://cdn.dierchat.ru/stories/thumbs/...",
  overlays: [ // текст, стикеры, рисунки
    { type: "text", data: {...} },
    { type: "sticker", data: {...} },
    { type: "drawing", data: {...} }
  ],
  privacy: "all" | "contacts" | "close_friends" | "custom",
  allowedUsers: [], // если privacy = "custom"
  excludedUsers: [], // если privacy = "all" но есть исключения
  createdAt: 1742563200,
  expiresAt: 1742649600, // +24 часа
  views: [
    { userId: "user_111", viewedAt: 1742563300, reaction: "❤️" },
    { userId: "user_222", viewedAt: 1742563350, reaction: null }
  ]
}
```

334. **API эндпоинты:**
    - `POST /stories/create` — создание истории.
    - `PUT /stories/:id` — редактирование истории.
    - `DELETE /stories/:id` — удаление истории.
    - `GET /stories/feed` — получить истории для ленты (для подписчиков).
    - `GET /stories/:id/viewers` — список просмотревших.
    - `POST /stories/:id/view` — отметить просмотр.
    - `POST /stories/:id/reaction` — поставить реакцию.

335. **WebSocket события:**
    - `story_new` — новая история у подписчика.
    - `story_deleted` — история удалена (обновить интерфейс).
    - `story_viewed` — кто-то посмотрел историю (для автора).
    - `story_reaction` — кто-то поставил реакцию.

## 29.11 Ограничения и лимиты

339. **Лимиты на истории:**

    - Видео: до 1080p, битрейт до 2.5 Mbps.

РАЗДЕЛ 31: Сворачивание окна звонка и видеозвонка (мини-окно)

При сворачивании звонка (кнопка «Свернуть» или клик за пределы) окно превращается в мини-панель (pip — picture-in-picture).
Для обычного аудиозвонка: мини-окно показывает аватарку + имя собеседника + таймер звонка + кнопки «Развернуть / Завершить».
Для видеозвонка: мини-окно показывает живую камеру говорящего (или демо экрана, если включена трансляция).
Мини-окно всегда «плавает» поверх всех чатов (position: fixed, z-index выше всего).
На ПК — можно перетаскивать мини-окно. На телефоне — прилепляется к краю экрана.
При сворачивании продолжать писать в любом чате без прерывания звонка.
Реализация: WebRTC track остаётся активным, видео/аудио рендерится в отдельный <video> элемент 120×120 px (или 200×120 px для демо).

Как в Telegram: точно такое же мини-окно (pip) с камерой говорящего.
РАЗДЕЛ 32: Исправление демонстрации экрана (демки)

Другой пользователь теперь видит трансляцию экрана в реальном времени (было: только у звонящего).
Использовать getDisplayMedia + добавление трека в существующий peerConnection.
Поддержка: весь экран / окно приложения / вкладка браузера (на ПК).
В группах — демка отображается как отдельный «участник» с лейблом «Демонстрация экрана».
Автоматическое переключение: если звонящий переключил источник — все видят новый.
Качество: адаптивный битрейт (до 1080p/30 fps).

РАЗДЕЛ 33: Быстрый переход на чат на десктопе (широкий экран)

При открытом окне «Настройки» (или любой боковой панели) клик по любому чату в левом списке автоматически закрывает настройки.
На горизонтальном десктопе (>1024 px) настройки занимают правую треть экрана — клик по чату слева мгновенно заменяет контент.
Анимация: плавное «сдвигание» панели настроек в сторону (translateX).
Нет необходимости нажимать кнопку «Назад».

РАЗДЕЛ 34: Демки и вебки в группах (правильный формат)

Камера и демонстрация экрана в групповых звонках отображаются в оригинальном соотношении сторон (не растягиваются и не обрезаются).
Для демки экрана — формат 16:9, для веб-камеры — 4:3 или 9:16 (автоопределение).
В группе одновременно могут быть: камеры участников + одна активная демка.

РАЗДЕЛ 35: Полная починка аватарок

Аватарки теперь отображаются везде: список чатов, шапка чата, реакции, уведомления, мини-окно звонка, профили.
При установке аватарки — открывается кроппер (cropper.js или встроенный canvas): пользователь выбирает область (круглая маска 1:1).
Масштабирование: аватарки 40×40 (список), 48×48 (шапка), 120×120 (профиль) — без искажений.
Кэширование через IndexedDB + WebSocket-обновление при смене.

РАЗДЕЛ 36: Адаптивные быстрые ответы

Панель быстрых ответов («Ок», «Спасибо», «Перезвоню» и т.д.) теперь можно полностью отключить в настройках → «Чаты» → «Быстрые ответы».
На телефоне панель выезжает только при фокусе поля ввода (адаптив).
На ПК — всегда скрыта, если отключена.

РАЗДЕЛ 37: Статус в сети в реальном времени

Зелёная точка «онлайн» обновляется мгновенно через WebSocket (событие online_status).
«Был(а) в сети X минут назад» — точное время с сервера.
Работает везде: список чатов, шапка, профиль.

РАЗДЕЛ 38: Всплывающие уведомления внутри приложения

При новом сообщении — toast-уведомление в правом нижнем углу (ПК) или сверху (телефон).
Клик по toast → мгновенный переход в чат.
Настройка: звук + вибрация + отключение для отдельных чатов.

РАЗДЕЛ 39: Исправление встроенного браузера (iframe-блокировка)

Проблема: Google и многие сайты блокируют iframe (чёрный экран).
Решение №1 (встроенный): использовать Service Worker + proxy-страницу (обход CSP).
Решение №2 (рекомендуемое): при попытке открыть такой сайт показывать кнопку «Открыть в системном браузере» + «Открыть в отдельном клиенте Dier Chat Browser».

После завершения — «Звонок завершён».

**РАЗДЕЛ 46: ПРАВИЛЬНАЯ ОТПРАВКА И ФОРМАТИРОВАНИЕ ТЕКСТОВЫХ СООБЩЕНИЙ В МЕССЕНДЖЕРЕ (СОХРАНЕНИЕ СТРУКТУРЫ ТЕКСТА)**

**Цель:**  
При отправке любого текста (обычное сообщение, подпись к медиа, био в профиле, название группы/канала, быстрый ответ и т.д.) структура текста должна сохраняться **точно в том виде, в каком пользователь его ввёл** — без потери переносов строк, пробелов, табуляций, без добавления мусора из символов, без автоматического удаления пустых строк, без лишних экранирований и без преобразования в HTML/ Markdown принудительно.

**Текущая проблема (наблюдаемая в 2026 году):**  
При отправке многострочного текста (например, списка, стиха, кода, инструкции) в мессенджере происходят следующие искажения:  
- несколько переносов строк сжимаются в один  
- начальные/конечные пробелы удаляются  
- табуляции превращаются в 4 пробела или вообще пропадают  
- иногда добавляются случайные невидимые символы (zero-width space, BOM, неразрывные пробелы)  
- при копировании/вставке из других приложений структура полностью ломается  

**Требования к реализации (46.1–46.15):**

46.1. Текст, введённый пользователем в поле ввода, должен отправляться на сервер **в точности как есть** (raw string), без какой-либо нормализации или санитизации на фронтенде.

46.2. На фронтенде (поле ввода):  
- Использовать `<textarea>` вместо `<div contenteditable>` (или аналог с `white-space: pre-wrap;`).  
- Сохранять все символы: `\n`, `\t`, пробелы, множественные переносы.  
- Запретить автоматическое преобразование в Markdown/HTML на этапе ввода.

46.3. При отправке (API):  
- Передавать поле `text` как чистую строку (UTF-8).  
- Не применять `trim()`, `replace(/\s+/g, ' ')` и подобные функции.  
- Не экранировать символы (кроме случаев, когда это требует протокол безопасности, например, `<script>` — но только на этапе отображения, а не хранения).

46.4. На сервере:  
- Сохранять текст в базе данных **как есть** (TEXT или VARCHAR с кодировкой utf8mb4).  
- Не применять функции нормализации строк (strip_tags, htmlspecialchars и т.п.) при записи.  
- Хранить оригинальную строку без изменений.

46.5. При получении и отображении сообщения:  
- Использовать CSS:  
  ```css
  .message-text {
    white-space: pre-wrap;      /* сохраняет переносы и пробелы */
    word-wrap: break-word;      /* длинные строки переносятся */
    overflow-wrap: break-word;
    font-family: system-ui, -apple-system, monospace; /* для кода лучше читаемость */
  }
  ```
- Не применять `innerHTML` с автоматическим парсингом Markdown, если пользователь не включил форматирование вручную (например, через **жирный** или `код`).

46.6. Поддержка форматирования (опционально):  
- Если пользователь хочет Markdown — добавить кнопку «Форматировать» (жирный, курсив, код, цитата).  
- Но по умолчанию — **отключить** любой автоматический Markdown-парсинг.  
- Хранить два поля: `raw_text` (оригинал) и `formatted_text` (если включено).

46.7. Копирование текста:  
- При долгом нажатии на сообщение → «Копировать» должен копировать **оригинальный raw-текст** (с переносами и пробелами).  
- Использовать `navigator.clipboard.writeText(message.raw_text)`.

46.8. Черновики и редактирование:  
- При сохранении черновика — хранить точную строку (localStorage или IndexedDB).  
- При редактировании отправленного сообщения — редактировать raw-текст, а не отформатированную версию.

46.9. Тестирование структуры (обязательные кейсы):  
- 5 пустых строк подряд → должны остаться 5 пустых строк  
- Текст с табуляцией (4 пробела или \t) → табуляция сохраняется  
- Список с отступами (пробелы перед номерами) → не сжимается  
- Код с отступами (Python, JS) → читаем как есть  
- Текст, скопированный из Word/Notepad → сохраняет все переносы  

46.10. Запрет на добавление мусора:  
- Запретить вставку zero-width space, BOM, неразрывных пробелов при отправке (можно фильтровать только эти символы на фронте, если они не нужны).  
- При получении текста от сервера — не добавлять никаких символов.

46.11. Пример правильного поведения (ожидаемый результат):

Пользователь ввёл:
```
Привет!

Это список:
    1. пункт один
    2. пункт два

    Код:
        print("Hello")
```

После отправки и получения должно отобразиться **точно так же**, без потери отступов и пустых строк.

46.12. Интеграция с быстрыми ответами:  
- Шаблоны («Ок», «Спасибо») отправляются как одна строка, но если пользователь добавил переносы — они сохраняются.

46.13. Логирование для отладки:  
- При отправке логировать длину строки и количество `\n` (чтобы разработчик видел, что структура не теряется).

46.14. Совместимость со старыми клиентами:  
- Если старый клиент получает сообщение с переносами — он должен хотя бы не ломать их при отображении (даже если не поддерживает pre-wrap).

46.15. Документация для разработчиков:  
- Добавить в README или в код комментарий:  
  «ВАЖНО: Текст сообщений хранится и передаётся как raw string. НЕ ПРИМЕНЯТЬ trim(), normalize(), replace(/\s+/g) при записи/отправке. Использовать white-space: pre-wrap при отображении.»

## P0 (март 2026): «Чат / 0 участников» и пустая лента сообщений — формулировка для ТЗ

**Симптомы (как видит пользователь)**  
1. В шапке открытого чата отображается заголовок **«Чат»** и подпись вроде **«0 участника»**, хотя в информации о чате (клик по шапке) всё верно: аватар, ник, состав группы/канала.  
2. Отправленные сообщения **не появляются** в ленте; после выхода из чата или перезапуска приложения **история пустая** (хотя на сервере сообщения могут сохраняться).

**Корневая причина (типичная)**  
Несовпадение строкового представления **одного и того же UUID** чата в разных местах клиента: регистр символов, наличие/отсутствие дефисов, разные источники id (список диалогов, push, поле `chat_id` в сообщении, persist). В коде чат из списка ищется как `chats.find(c => c.id === currentChatId)` — при любом отличии строк объект чата **не находится** (`chat === null`), ломается тип чата (личка/группа/канал) и счётчики, при этом запросы `GET /messages` и `GET /members` по **тому же** id часто продолжают работать, поэтому панель информации может быть корректной.

**Требования к исправлению**  
1. **Канонизация UUID** на клиенте: один формат (нижний регистр, с дефисами) для `currentChatId`, `chat.id`, `message.chat_id`, `peer_user_id` и т.д.  
2. **Поиск чата в списке** по каноническому сравнению, не через «сырое» `===`.  
3. **Нормализация ответов API** списка чатов и сообщений (приведение id и числового `type`).  
4. **WebSocket `new_message`:** если в теле сообщения нет `chat_id`, подставлять `chat_id` из корня события.  
5. Сборка под продакшен: `.env.production` с `VITE_API_BASE_URL` (и при необходимости `VITE_WS_URL`); скрипты `npm run build:host` / `npm run package:host` (см. `DierCHAT-Desktop`).

**Критерии приёмки**  
- В личном чате в шапке — имя/username собеседника и корректный статус, не «Чат» и не «0 участника» без причины.  
- В группе/канале — название и актуальное число участников/подписчиков (с учётом `member_count` с сервера, если список участников ещё не подгрузился).  
- Сообщение видно сразу после отправки (optimistic UI) и после перезахода в чат / перезапуска приложения.  
- Событие `new_message` обновляет ленту для текущего чата.
